توجه

کتابخونه های جاوا در زبان برنامه نویسی کاتلین هم قابل استفاده هستن و این مطلب از این قائده مستثنی نیست بنابراین مثال های این مطلب علاوه بر جاوا با کاتلین هم زده شده است؛ و باید به این نکته هم توجه کرد که کاتلین علاوه بر اینکه از کتابخونه های چندنخی جاوا میتونه استفاده کنه، یک کتابخونه ی اختصاصی براش وجود داره به نام کروتینز (Coroutines) که بر اساس چندنخی در جاوا پیاده سازی شده است.

مفهوم رقابت بین نخ ها (race condition) و همگام سازی (synchronizing)

زمانی که چند نخ به طور همزمان بخوان یک داده رو تغییر بدن، هنگام ایجاد تغییرات روی داده ممکنه باعث خراب شدن داده بشن و اصطلاحا میگیم data corruption رخ داده، فرض کنید میخوایم یک متغیر رو با دو عدد به طور همزمان جمع کنیم، برنامه نمیدونه کدوم عدد رو اضافه کنه و ممکنه به طور تصادفی عدد اول رو با متغیر جمع کنه و از عدد دوم صرف نظر کنه، شایدم برعکس عدد دوم رو با متغیر جمع بزنه، به این وضعیت در چندنخی (multithreading) وضعیت رقابت بین نخ (Thread) ها (race condition) میگیم.

همگام سازی با کلیدواژه ی synchronized

همگام سازی نخ یعنی هماهنگ کردن اجرای نخ هایی که به هم وابسته هستن.

برای جلوگیری از race condition یک کلیدواژه به نام synchronized در جاوا وجود داره، هنگامی که از این کلیدواژه استفاده بشه، تنها یک نخ میتونه به داده دسترسی داشته باشه و تا زمانی که کار نخ (thread) تموم نشده داده قفل میمونه و نخ های دیگه به داده دسترسی ندارن.

از کلیدواژه ی synchronized میتونیم برای آبجکت و متد (تابع) استفاده کنیم.

توجه

در کاتلین کلیدواژه ی synchronized وجود نداره، برای همگام سازی از کلاس ReentrantLock استفاده میکنیم؛ کلاس ReentrantLock یک پیاده سازی در جاواست، معمولا از ReentrantLock برای کنترل بیشتر روی همگام سازی داده ها با نخ ها استفاده میشه، بنابراین در این قسمت از مطلب مثال از کاتلین نداریم و مثال های کاتلین از قسمت ReentrantLock شروع میشه.

استفاده از کلیدواژه ی synchronized برای آبجکت

با استفاده از کلیدواژه ی synchronized میتونیم یک قفل روی آبجکت ایجاد کنیم تا هنگام استفاده از آبجکت توسط چند نخ (Thread) دچار race condition نشیم.

فرم کلی:

synchronized (expression){
   //statement
}

بجای expression متغیری که به آبجکت اشاره می کنه رو می نویسیم و داخل بلوک توابع یا فیلد های آبجکت رو که میخوایم قفل بشن صدا میزنیم.

در مثال قبل میتونیم از synchronized برای متغیر account استفاده کنیم و از race condition جلوگیری کنیم.

private static class IncrementScoreByOne implements Runnable{
    @Override
    public void run() {
        synchronized (account) {
            account.updateScore(1);
        }
    }
}

استفاده از کلیدواژه ی synchronized برای متد

تابع برای نخ های دیگه ازاد نمیشه و نخ های دیگه نمیتونن تابع رو اجرا کنن، به متد هایی که با کلیدواژه ی synchronized تعریف میشن، متد های همگام سازی شده (Synchronized Methods) گفته میشه.

در مثال قبل برای اینکه race condition به وجود نیاد کافیه متد (تابع) updateScore رو با کلیدواژه ی synchronized تعریف کنیم.

private static class Account{
  private Integer score = 0;
  public synchronized void updateScore(int v){
      int newScore = score + v;
      try {
          Thread.sleep(5);
      }catch (InterruptedException e){
          throw new RuntimeException(e);
      }
      score = newScore;
  }
  public int getScore() {
      return score;
  }
}

از این به بعد زمانی که گفته میشه یک متد در جاوا synchronized است باید بدونید منظور چیه :)

همکاری بین نخ (Thread) ها با متد های wait, notify و notifyAll

نخ (thread) ها میتونن باهمدیگه همکاری کنن، دوتا نخ رو در نظر بگیرید زمانی که اجرای نخ اول به جایی برسه، نخ اول متوقف بشه و نخ دوم شروع به کار کنه و زمانی که اجرای نخ دوم به جایی برسه، نخ دوم متوقف بشه و نخ اول شروع به کار کنه.

برای بدست اوردن چنین عملکردی در کلاس Object جاوا سه متد (تابع) به نام های wait, notify و notifyAll تعریف شده، از این سه تابع میتونیم داخل متد هایی که با synchronized تعریف شده باشن یا عبارت synchronized استفاده کنیم.

توجه

استفاده از سه متد wait, notify و notifyAll فقط در جاهایی که synchronized تعریف شده امکان پذیر است.

متد (تابع) wait

از این متد (تابع) برای متوقف کردن نخ هایی که دارن از آبجکت مورد نظر استفاده میکنن، استفاده کنیم؛ و برای ادامه ی کار نخ منتظر سیگنال ادامه میمونه.

متد (تابع) notify

اگه یک نخ توسط یک آبجکت با wait متوقف شده باشه، با استفاده از متد notify میتونیم سیگنال ادامه رو به نخ (thread) بدیم. سیگنال ادامه ی کار فقط به نخ هایی داده میشه که توسط همون آبجکت با wait متوقف شده باشن، و اگه چند نخ توسط آبجکت با wait متوقف شده باشه با notify فقط یکیشون به کار ادامه میده.

متد (تابع) notifyAll

از این متد برای دادن سیگنال به تمام نخ های متوقف شده روی آبجکت فعلی استفاده می کنیم، استفاده از این متد هر نخی که روی این آبجکت متوقف شده باشه رو اجرا می کنه، سیگنال ادامه ی کار به نخ هایی داده میشه که توسط همون آبجکت با wait متوقف شده باشن و تمام نخ های متوقف شده با سیگنال notifyAll به کار خودشون ادامه میدن.

مثال زیر یک برنامه ی تولید کننده و مصرف کننده است (Consumer-Producer) یک مثال کلاسیک برای درک مفهوم همکاری بین نخ (thread) ها؛ در این مثال یک کلاس داریم و دوتابع به اسم produce و consume، زمانی که مقدار data صفر باشه تابع consume متوقف میشه و سیگنال ادامه ی کار رو به produce می فرسته و تابع produce شروع به کار میکنه تا مقدار داده تغییر کنه، زمانی که مقدار داده تغییر کرد، تابع produce متوقف میشه و سیگنال ادامه ی کار رو به تابع consume میفرسته و consume شروع به ادامه ی کار میکنه.

import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class ConsumerProducerDemo {

    public static void main(String[] args) {
        ExecutorService service = Executors.newFixedThreadPool(2);

        ConsumerProducer cp = new ConsumerProducer();

            service.execute(() -> {
                while (true)
                    cp.consume();
            });


            service.execute(() -> {
                while (true)
                    cp.produce();
            });

            service.close();

    }

    private static class ConsumerProducer{
        private int data;


        public synchronized void consume(){
            try {

                while (data == 0) {
                    wait();
                }
                Thread.sleep(1000);
            }catch (InterruptedException e){
                e.printStackTrace();
            }

            System.out.println("Consuming " + data);
            data = 0;
            notify();
        }

        public synchronized void produce(){

            try {
                while (data != 0) {
                    wait();
                }

                Thread.sleep(2000);
            }catch (InterruptedException e){
                e.printStackTrace();
            }

            data = (int) (1 + (Math.random() * 126));
            System.out.println("Producing " + data);
            notify();
        }
    }
}

همگام سازی (synchronizing) با کلاس ReentrantLock

با استفاده از Lock و condition میتونیم به صورت آشکار (explicitly) داده ها رو با نخ ها همگام سازی کنیم.

با کلیدواژه ی synchronized یک قفل پنهانی (implicit lock) روی نمونه قرار میدادیم و هنگام استفاده از متد توسط نخ (thread)، متد برای استفاده از نخ قفل میشد.

با استفاده از کلاس ReentrantLock میتونیم بدون نیاز به کلیدواژه ی synchronized یک متد رو به صورت آشکار (explicitly) قفل کنیم؛ یکی از پیاده سازی های داخل کلاس ReentrantLock، اینترفیس Lock است که متد های پرکاربرد lock, unlock و newCondition داخل اینترفیس lock تعریف شده اند.

توجه

در اکثر موارد استفاده از کلیدواژه ی synchronized کفایت میکنه ولی استفاده از ReentrantLock کنترل بیشتری روی هماهنگ کردن نخ ها به برنامه نویس میده.

در زیر نمودار uml ارتباط ReentrantLock با اینترفیس Lock و متد های پیاده سازی شده ی داخلش شرح داده شده است.

java.util.concurrent.locks.Lock
+lock(): void یک قفل ایجاد میکنه.
+unlock(): void قفل رو آزاد میکنه.
+newCondition(): Condition یک نمونه از Condition رو بر میگردونه.

اینترفیس Lock در کلاس ReentrantLock پیاده شده است.

java.util.concurrent.locks.ReentrantLock
+ReentrantLock() یک نمونه از کلاس ReentrantLock ایجاد میکنه.
+ReentrantLock(fair: boolean) یک نمونه از کلاس ReentrantLock ایجاد میکنه و زمانی که مقدار پارامتر true باشه و نخی که بیشتر از بقیه ی نخ ها تو صف استفاده از داده قرار گرفته، اولین اولویت استفاده از داده بعد از آزاد شدن داده رو داره.

اینترفیس Condition

هنگام استفاده از کلاس ReentrantLock بجای متد های wait، notify و notifyAll باید از متدهای اینترفیس Condition استفاده کنیم، برای ایجاد یک نمونه از condition کافیه متد newCondition رو با ReentrantLock صدا بزنیم.

<<java.util.concurrent.Condition>>
+await(): void باعث میشه نخ (thread) فعلی به طور موقت متوقف بشه تا وقتی که سیگنال ادامه بهش داده بشه.
+signal(): void به یکی از نخ (thread) های متوقف شده با await سیگنال اجرا میده.
+signalAll(): void به تمام نخ (thread) های متوقف شده با await سیگنال اجرا میده.

میخوایم مثال ConsumerProducer رو با ReentrantLock پیاده کنیم.

public class ConsumerProducerReentrantLock {
    public static void main(String[] args) {

        ConsumerProducer cp = new ConsumerProducer();

        ExecutorService service = Executors.newFixedThreadPool(2);

        service.execute(() -> {
            while (true)
                cp.consume();

        });
        service.execute(() -> {
            while (true)
                cp.produce();

        });

        service.close();
    }


    private static class ConsumerProducer{

        private int data = 0;

        private final Lock lock = new ReentrantLock();
        private final Condition consumeCondition = lock.newCondition();
        private final Condition produceCondition = lock.newCondition();


        public void consume(){

            try {
                lock.lock();
                while (isEmpty()) {
                    consumeCondition.await();
                }

                Thread.sleep(1000);

                System.out.println("consuming " + data);
                data = 0;
                produceCondition.signal();

            }catch (InterruptedException e){
                e.printStackTrace();
            }finally {
                lock.unlock();
            }
        }


        public void produce(){

            lock.lock();

            try {
                while (!isEmpty()){
                    produceCondition.await();
                }

                Thread.sleep(2000);

                data = (int)(Math.random() * 255);
                System.out.println("producing " + data);
                consumeCondition.signal();

            }catch (InterruptedException e){
                e.printStackTrace();
            }finally {
                lock.unlock();
            }
        }

        private boolean isEmpty(){
            return data == 0;
        }
    }
}

محدود کردن دسترسی تعداد نخ (thread) ها به داده با کلاس Semaphore

با استفاده از کلاس Semaphore در جاوا میتونیم تعداد نخ هایی که به داده میتونن دسترسی داشته باشن رو محدود کنیم، برخلاف synchronized و ReentrantLock که فقط به یک نخ دسترسی به داده رو میدن با Semaphore میتونیم دسترسی به داده رو محدود به یک یا چند نخ کنیم، به عبارتی خودمون تعیین میکنیم که چندتا نخ همزمان میتونه به یک داده دسترسی داشته باشه.

java.util.concurrent.Semaphore
+Semaphore(numberOfPermits) یک نمونه از Semaphore ایجاد می کنه و داخل پارامتر کانستراکتور تعداد نخ هایی که میتونن از داده ها (منابع) استفاده کنن رو تعیین میکنیم.
+Semaphore(numberOfPermits: int, fair: boolean) یک نمونه از Semaphore ایجاد میکنه و تعداد نخ هایی که میتونن از داده (منابع) استفاده کنن رو تعیین میکنیم و همینطور با boolean مشخص میکنیم نوبت دهی منصفانه باشه یعنی نخی که بیشتر از همه منتظر مونده تا از منابع استفاده کنه زمانی که کار یکی از نخ ها تموم بشه اولویت بالاتری نسبت به بقیه داره تا شروع کنه.
+acquire(): void محدودیت استفاده از داده (منابع) رو ایجاد میکنه.
+release(): void محدودیت ایجاد شده رو بر میداره.

مفهوم Deadlock

معلم میره سر کلاس میبینه دانش آموز نیست، دانش آموز میره سر کلاس میبنه معلم نیست، این وضعیت تا زمانی که ادامه پیدا کنه مدرسه تعطیله، بحث Deadlock مشابه همینه دوتا کار در دو نخ جدا به همدیگه وابسته ان، هردو قفل شدن و هرکدوم منتظر اون یکیه تا شروع بشه یا ادامه پیدا کنه.

در مثال زیر از sleep داخل عبارت synchronized مربوط به object استفاده کردیم تا Deadlock شکل بگیره.

ExecutorService service = Executors.newFixedThreadPool(2);

Object object = new Object();
Object object1 = new Object();


service.execute(() -> {
  synchronized (object) {

      System.out.println("From object");

      try {
          Thread.sleep(200);
      } catch (InterruptedException e) {

      }

      synchronized (object1) {
          System.out.println("Causes dead lock");
      }
  }
});


service.execute(() -> {

  synchronized (object1) {
      System.out.println("From object1");

      synchronized (object) {
          System.out.println("Causes dead lock");
      }
  }
});

service.close();

یکی از کار هایی که ممکنه باعث deadlock بشه صدا زدن notify یا notifyAll توسط یک آبجکت دیگه برای ادامه ی اجرای thread یک آبجکت دیگس.

فرض کنید در مثال ConsumerProducer فرض کنید بجای یک کلاس مشترک برای متد های consume و produce یک کلاس جدا برای consume تعریف کرده باشیم و یک کلاس دیگه برای produce و مانند قسمت قبل میخوایم از wait و notify برای توقف و سیگنال دادن به نخ هاشون استفاده کنیم.

public class ConsumerProducerDemo {

    public static void main(String[] args) {
        ExecutorService service = Executors.newFixedThreadPool(2);

        Producer producer = new Producer();
        Consumer consumer = new Consumer();

        //ConsumerProducer cp = new ConsumerProducer();

            service.execute(() -> {
                while (true)
                    consumer.consume(producer);
            });


            service.execute(() -> {
                while (true)
                    producer.produce();
            });

            service.close();

    }


    private static class Producer{
        private int data ;

        public synchronized void produce(){
            try {
                while (data != 0)
                    wait();
            }catch (InterruptedException e){
                e.printStackTrace();
            }

            data = (int) (1 + (Math.random() * 126));
            System.out.println("Producing " + data);
            notifyAll();

        }

        public void resetData(){
            data = 0;
        }
    }

    private static class Consumer{

        public synchronized void consume(Producer producer){
            try {

                while (producer.data == 0) {
                    wait();
                }
                Thread.sleep(1000);
            }catch (InterruptedException e){
                e.printStackTrace();
            }

            System.out.println("Consuming " + producer.data);
            producer.resetData();
            notifyAll();
        }
    }

}

برای جلوگیری از deadlock از روش ترتیب منابع (resource ordering) استفاده کنید هر چیزی که قفل روشه رو به ترتیب با نخ (thread) اجرا کنید و از اجرای قفل های تو در تو پرهیز کنید و همینطور موقع صدا زدن wait و notify دقت کنید که این متد ها پس زمینه ی آبجکتی باشن که داریم از متد هاش استفاده می کنیم.

خلاصه

  • زمانی که چندتا نخ به صورت همزمان دارن از یک داده استفاده میکنن، race condition پیش میاد.
  • برای جلوگیری از race condition در جاوا میتونیم از کلیدواژه ی synchronized یا از کلاس های ReentrantLock و Semaphore استفاده کنیم.
  • زمانی که از کلاس ReentrantLock استفاده میکنیم بجای wait, notify و notifyAll باید از متد های await, signal و signalAll در اینترفیس condition استفاده کنیم.
  • با استفاده از متد newCondition در Lock میتونیم یک نمونه از Condition استفاده کنیم.
  • با استفاده از کلاس Semaphore میتونیم تعداد نخ هایی که میتونن به یک داده دسترسی داشته باشن رو محدود کنیم.
